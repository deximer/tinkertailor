1. Executive Summary

Tinker Tailor (TT) is an AI-powered fashion design platform that enables consumers and creators to design custom garments through natural language conversation, visualize them in real-time 3D, and have them manufactured on-demand. The platform connects a structured pattern library, an AI design assistant (TARA), 3D rendering, e-commerce, influencer revenue sharing, and factory fulfillment into a single end-to-end experience.

This brief defines the technical architecture, technology choices, deployment strategy, and MVP scope for the initial build. It is intended as a companion to the Engineering Services Agreement and establishes shared understanding of what will be built, how, and in what order.

2. MVP Definition

2.1 Scope

The MVP is a closed, invite-only pilot with 10–15 influencer/creator accounts producing capsule collections. Their followers and other invited users will have full access to the platform as shoppers—browsing designs, using the design tools, and purchasing garments. The platform is expected to support thousands of concurrent shoppers during the pilot period. The goal is to prove the core design loop end-to-end:

Design it → Post it → Share it → Sell it → Ship it

  • Cornate color printing (hex-code fabric coloring)
  • Tech pack download / purchase
  • Photo-realistic rendering from 3D models
  • Multi-factory routing / supply chain telemetry
  • Image upload for AI pattern matching
  • Personalized body avatars / virtual try-on
  • Promotional credits system
  • Embellishments (buttons, zippers, decorative items)—layering on 3D renderings
  • AI usage credits / metering (free tier initially; usage-based pricing to be defined)
  • Influencer stream scanning and aesthetic-aligned style suggestions
  • AI-driven pattern adjustments without human involvement
  • Jumpsuit and additional garment category support beyond dresses, tops, and skirts

2.4 Constraints & Assumptions

  • 3D renderings are pre-created in Clo 3D by an external artist and ingested into the system; the app does not generate or modify 3D models at runtime
  • Fabric rendering on the front end is texture overlay on existing 3D assets, not real-time physics simulation
  • Component-level assembly composites pre-rendered 3D parts; the app does not procedurally generate new 3D geometry from component combinations
  • A Python codebase from the original Tinker Tailor application will be made available for analysis; this contains the prior silhouette composition logic and will serve as a reference for implementing component assembly, compatibility validation, and related business rules in the new system
  • The pattern library ships with a finite catalog (~300 components) with pre-defined compatibility rules
  • Factory integration targets a single MVP factory using iCreate; multi-factory routing is post-MVP
  • Color is constrained to available factory fabric stock for MVP (no on-demand printing)

Why Not Vercel?

Vercel was evaluated and is the default deployment target for Next.js applications. It offers excellent developer experience with zero-config CI/CD, automatic preview deployments, and native framework integration. However, it was decided against for the following reasons:

  • Serverless ceiling: Vercel’s serverless model imposes function duration limits (60 seconds on Pro) that constrain LLM orchestration with multi-step tool-calling, factory order assembly, and webhook processing chains. The platform’s post-MVP roadmap—background job processing, multi-factory routing, real-time collaboration—pushes further beyond what stateless functions support well.
  • Replatforming risk: Vercel deploys code into a proprietary runtime that splits the application across edge functions, serverless functions, and static assets. Migrating off Vercel requires reassembling these pieces into a conventional application—an engineering project, not a configuration change. With Fly.io, the application runs in a standard Docker container that can be deployed to AWS ECS, GCP Cloud Run, or any container platform with minimal effort.
  • Portability: A Docker-based deployment ensures the application is never locked to a single hosting provider. If infrastructure needs scale massively post-MVP, migration to AWS, GCP, or multi-cloud is a DevOps task (days) rather than an application rewrite (weeks).

Path to AWS

Fly.io is the right platform for MVP velocity, but the architecture is designed so that migrating to AWS is straightforward if scale or compliance requirements demand it. Because the application runs as a standard Docker container, the migration path is:

  1. Push the same Docker image to AWS Elastic Container Registry (ECR)
  2. Run it on ECS Fargate (serverless containers) or EKS (Kubernetes)—no application code changes required
  3. Replace Fly.io secrets with AWS Secrets Manager or Parameter Store
  4. Swap the GitHub Actions deploy step from fly deploy to aws ecs update-service
  5. Add an Application Load Balancer for SSL termination and routing

Supabase (database, auth, storage) remains unchanged—it connects over standard PostgreSQL and HTTPS regardless of where the application runs. The total migration effort is measured in days of DevOps work, not weeks of engineering, and can be executed without modifying application code.

3.2 Infrastructure Sizing

The MVP pilot targets 10–15 influencer/creator accounts but expects thousands of shoppers (creators’ followers and other invitees). The application runs as a containerized process on Fly.io and scales horizontally by adding machines in additional regions as demand requires:

  • Fly.io: Shared-CPU machines initially, scaling to dedicated-CPU as traffic warrants; multiple machines across regions for availability and latency; no function invocation limits or duration ceilings
  • Supabase: Pro plan minimum; connection pooling enabled for concurrent database access; storage sized for 3D asset catalog
  • Asset Delivery: 3D renderings and fabric textures served from Supabase Storage
  • Database: Query optimization and indexing for the compatibility graph, tag resolution, and product catalog queries under concurrent load

If traffic volume grows beyond what Supabase Storage handles efficiently, a CDN layer (e.g., Cloudflare, AWS CloudFront) can be introduced in front of asset storage to reduce latency and offload bandwidth. This is a configuration change, not an architectural one.

Infrastructure costs scale with user volume. Hosting and third-party service costs are Client’s responsibility and are not included in the project fee.

3.3 Scaling Strategy

The platform launches with a minimal footprint and scales incrementally as traffic data dictates. Each lever below is independent—pull the one that addresses the actual bottleneck rather than scaling everything at once.

Vertical Scaling — Bigger Machines

The simplest first move. Fly.io machine size is a single configuration change with no downtime:

Machine TypeRAMUse Case
shared-cpu-1x2GBDevelopment / staging
shared-cpu-2x4GBMVP launch
dedicated-cpu-2x8GBSustained concurrent load
dedicated-cpu-4x16GBHeavy traffic / LLM processing

Horizontal Scaling — More Machines

Run multiple instances of the application simultaneously. Fly.io load-balances across them automatically. Because Next.js is stateless—session state lives in Supabase, design state in the browser—additional machines require no application changes:

  • Scale from 1 to N machines with a single command (fly scale count N)
  • Enable autoscaling to handle traffic spikes automatically—machines start in ~2 seconds and spin down during quiet periods; keep at least one machine warm at all times to avoid cold starts

Geographic Scaling — More Regions

Deploy machines in additional Fly.io regions to reduce latency for international users. Fly routes each request to the nearest healthy machine. Relevant as the creator/shopper base grows outside the initial launch market.

Service Splitting — Isolate Heavy Work

As backend complexity grows, resource-intensive workloads can be moved to dedicated Fly Machines that scale independently from the main web application:

ServiceRationale
TARA / LLM workerLLM calls are slow and CPU-heavy; isolating them prevents blocking web request handling
Background job workerFactory order assembly, payout calculations, and ShipStation polling are long-running and should not compete with HTTP traffic

Database Scaling — Supabase

The database is often the first bottleneck under load. Steps in order of effort:

  • Connection pooling: Enabled from day one via Supabase’s built-in PgBouncer
  • Query optimization: Indexes on compatibility graph traversal, tag resolution, and order lookups before adding hardware
  • Read replicas: Route catalog browsing and read-heavy queries to replicas, reserving the primary for writes; available on Supabase Pro
  • Plan upgrade: More connections and compute via Supabase plan tier

Asset Delivery — CDN

If 3D asset delivery becomes a bandwidth or latency bottleneck, a CDN layer (Cloudflare or AWS CloudFront) can be placed in front of Supabase Storage. This is a configuration change with no application code impact.

LLM Orchestration: Vercel AI SDK vs LangChain.js

The SDK choice will be finalised at implementation. Both options are provider-agnostic and support streaming and tool calling. The deciding factor is TARA’s conversational requirements:

Vercel AI SDKLangChain.js
StreamingFirst-classSupported
Tool callingClean, TypeScript-nativeVerbose but flexible
Structured outputZod-native via generateObject()Available, less ergonomic
Conversation memoryDIY — you manage the messages arrayBuilt-in memory modules including summarization
Long conversation handlingManual windowing or summarizationConversationSummaryMemory built-in
ComplexityLowHigher
Next.js fitNativeWorks, not optimized

TARA requires multi-turn conversational continuity—users explore ideas across many messages, refining as they go, with earlier context (occasion, mood, constraints) informing later suggestions. For shorter conversations, passing the full message history each turn is sufficient and favors the Vercel AI SDK. If conversations grow long enough to approach context limits, LangChain’s built-in summarization memory becomes a meaningful advantage. The final choice will be made once the expected conversation depth is better understood.

5. Platform Capabilities

5.1 TARA: The AI Design Assistant

TARA is a conversational AI assistant that guides customers from a vague idea (“I need a dress for a New York winter wedding”) to a specific, manufacturable garment selection. She maps natural language to the structured tag taxonomy (13 categories: silhouette, length, waist, neckline, sleeve, mood, era, occasion, etc.), queries the pattern library for matching silhouettes and component combinations, validates against the compatibility graph, and presents 3D-rendered options. Users refine iteratively through conversation. TARA is powered by a pluggable LLM via the Vercel AI SDK; provider selection will be based on persona consistency and tool-calling quality.

5.2 Imagine & Atelier: Two Design Modes

Imagine is the conversational design mode—users describe what they want in natural language, TARA presents options, and the user refines through dialogue. Atelier is a manual design mode for users who want direct control—browsing and selecting components by type, with the compatibility graph filtering options in real-time. Users can switch freely between modes; the design state is shared.

5.3 Creator & Influencer Marketplace

Creators design garments, share them via social media with tracked attribution links, and earn 20% of net sales from purchases driven through their links. Stripe Connect handles marketplace-style payout splitting. A creator dashboard provides visibility into designs, shares, sales, and earnings.

5.4 E-Commerce & Checkout

The checkout handles millions of possible product variations (pattern × fabric × color × size) that cannot be pre-enumerated as SKUs. Products are assembled at checkout from the user’s design session, with dynamic pricing based on components and materials. Stripe’s Payment Intents API supports the custom checkout flow; Stripe Connect provides creator payout infrastructure.

5.5 Factory Integration & Fulfillment

On payment confirmation, the system assembles a factory-readable order (mapping the design to pattern files, fabric stock codes, sizing, and shipping), pushes it to the factory via iCreate, and tracks fulfillment through ShipStation. iCreate API format and factory order specification are client dependencies (see ESA Section 1.2).

5.6 Component Compatibility Graph

The compatibility graph is the central data structure: it defines which components (bodice, sleeves, skirt) can be combined, by what connection type (empire waist, natural waist, sleeve cap, etc.), and with which fabrics. It drives design validation, TARA’s suggestions, Atelier UI filtering, and factory order specifications. The graph ships pre-populated from the existing compatibility spreadsheet and legacy Python codebase.

7. Engineering Approach

Engineering will be iterative, focusing on the most challenging, highest-value capabilities first. Rather than a sequential phase plan with fixed deliverables and dates, the team will deliver working, demonstrable capabilities incrementally. Each delivery is an opportunity for the Tinker Tailor team to validate direction and provide feedback that shapes what comes next.

The highest-risk work comes first—TARA’s AI design experience, 3D component compositing, the compatibility graph, factory integration—because that is where early feedback has the most leverage. These are the capabilities that only the Tinker Tailor team can validate: Does TARA feel right with real pattern data? Does the 3D quality meet the bar? Does the factory pipeline produce a correct order? Getting working versions of these into the client’s hands early reduces risk and improves outcomes far more than completing lower-risk work (auth, admin, checkout) on a predetermined schedule.

The Tinker Tailor team’s role in this process is to provide signal—reacting to working software, identifying gaps, and steering priorities based on what they see. This produces a tighter feedback loop and a better product than a milestone-approval model.
